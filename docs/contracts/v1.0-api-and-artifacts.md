# V1.0 API and Artifact Contracts

V1.0 contract identifiers are exposed by `GET /version` and
`molecule_ranker.release.release_manifest()`.

| Contract | Identifier |
| --- | --- |
| API | `api.v1` |
| Artifacts | `artifacts.v1` |
| Integration data contracts | `data-contracts.v1` |
| Warehouse schema | `mr_warehouse_v1.0.0` |

## API Contract

The V1.0 API contract stabilizes:

- `/health`, `/ready`, `/version`, and `/metrics`.
- Hosted authentication and service account token routes.
- Project, artifact, review, experiment, Codex job, integration, webhook,
  admin, audit, export, deletion, and retention routes.
- Structured error responses with request IDs.
- Bearer-token auth for hosted routes and explicit RBAC enforcement.

`GET /version` must return the current package version, `1.9.0` for V1.9, plus
all four contract identifiers. Backward-compatible additive response fields are
allowed within the V1 contract family. Breaking route, auth, artifact, or
error-shape changes require a new contract identifier.

## Artifact Contract

V1.0 release artifacts must keep these properties:

- Artifact manifests include IDs, types, paths, SHA-256 hashes, sizes,
  timestamps, and provenance metadata.
- Candidate reports separate source-backed evidence, literature claims,
  generated molecule hypotheses, developability triage, experimental evidence,
  review decisions, and Codex outputs.
- Generated molecule artifacts label hypotheses as computational and never as
  validated actives.
- Codex artifacts stay under `codex_backbone` or review-assistant outputs and
  do not become evidence, assay results, generated molecules, review decisions,
  or scores.
- Export packages exclude caches, secrets, bearer tokens, API keys, hidden
  environment files, local Codex credentials, and full copyrighted article text.

## Integration and Warehouse Contracts

Integration imports and exports must keep source system, source record ID,
record type, external version, sync job ID, sync record ID, timestamps, raw
metadata, validation reports, and audit events.

Warehouse exports use schema contract `mr_warehouse_v1.0.0` for curated tables,
including candidates, generated molecules, targets, evidence, literature
claims, developability assessments, assay results, review decisions,
active-learning suggestions, sync jobs, and artifact manifests.
